# triple-review — トリプルレビュー (3体 × 発散 × 人間判断)

このスキルは比較デモの **4本目にして最終**。ここで見せたい教訓は **「レビュアーを増やせば質が上がる、は嘘」** ということ。3体それぞれが違う流派で重箱の隅をつつくので、議論はまとまらず、最終判断は人間に戻る。YouTubeでは "増やしすぎた時の失敗パターン" として機能させる。

## 前提

- `report/00_internal.md` が存在する。なければ「先に `/self-review <機能名>` を実行してください」と伝えて停止する。

---

## 実行手順

### Step 1 — イントロ表示 (YouTube視聴者向け)

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎬 Pattern 4 / 4 : トリプルレビュー (発散パターン)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【前提条件】
・Pattern 1 (self-review) が完了している
・report/00_internal.md が存在する

【実行内容】
1. 3体のサブエージェントを並行起動する
2. 3体それぞれに "異なる流派の重箱の隅" をつつかせる:
   ・Agent A: スタイル / 命名 / コメント密度 / 西部劇口調
   ・Agent B: 型の厳密性 / マイクロ最適化 / メモリ
   ・Agent C: テスト粒度 / エッジケース / ドキュメント表現
3. 互いにレビューを見せ合っても 合意に至らない
4. メインは 3体の不一致 を整理するだけで判断しない
5. report/03_triple.md に保存し、
   「どの意見を採用するか」をユーザーに判断委譲する

【期待される効果 (=今回は "反面教師" としての効果)】
✓ レビュアーを増やせば質が上がる、という誤解を壊す
✓ 流派が違う指摘は合成できない (重箱の隅は統合できない)
✓ AI駆動開発でも「人間の判断点」を残す必然性が体感できる

【このパターンのコスト】
・サブエージェント3体ぶんのトークン
・にもかかわらず、決定的な結論は得られない

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Step 2 — チームを作って3体を異なるキャラクターで並行起動する

まず `TeamCreate` で発散レビュー用チームを作る (3体間の SendMessage がチームメンバー名で撃てるようになる):

```
TeamCreate({
  team_name: "triple-review",
  agent_type: "review-coordinator",
  description: "トリプルレビュー: 異流派3体の発散レビュー"
})
```

続けて **同じ1メッセージ内で Agent ツールを3回呼び出して並列起動する**。3体すべてに `team_name: "triple-review"` を渡す。

3体それぞれ個性を強めに演出する。重箱の隅をつつく流派を意図的に分けるのがこのパターンの肝。

#### Agent A: `name: "stylist"` (スタイル警察)

```
あなたは "スタイル警察" です。コードの動作よりも 命名 / 一貫性 / コメント
密度 / ドキュメント表現 / 文字列の書き方 に執着します。西部劇テーマの
プロジェクトなので、レスポンス文の西部劇口調の徹底具合にも目を光らせてください。

重箱の隅をつつくスタンスで構いません。むしろ推奨します。
"機能としては動くが美意識として許せない" レベルの指摘を歓迎します。

手順:
1. git diff と CLAUDE.md と変更ファイル を読む
2. report/00_internal.md も読む
3. 以下のJSONで返答:

{
  "reviewer": "stylist",
  "school": "スタイル / 命名 / 表現",
  "pet_peeves": [...],  // 自分のこだわりポイント
  "findings": [
    {"severity": "critical|major|minor", "detail": "..."}
  ],
  "merge_verdict": "approve|request_changes|block",
  "non_negotiable": ["これだけは譲れない点"]
}

他の2人のレビュアー (type-hawk, test-nitpicker) と並行に動いています。
後でメインから彼らのレビューを送るので、そのときは自分の立場を
譲らず、ただし彼らの観点は "別軸" として認めるスタンスで返答してください。
```

#### Agent B: `name: "type-hawk"` (型とマイクロ最適化の鷹)

```
あなたは "型の鷹" です。TypeScript の any, unknown, 広すぎるユニオン、
暗黙の undefined, マイクロ最適化、不要な再レンダ、N+1 っぽい非同期、
アロケーション、に敏感です。それ以外 (命名、テスト、ドキュメント) には
ほとんど興味がありません。

重箱の隅をつつくスタンスでOK。
"実害はないが型として気に食わない" レベルも遠慮なく出してください。

手順と出力JSON形式は stylist と同じ (reviewer="type-hawk", school="型安全性 / マイクロ最適化")。
他の2人 (stylist, test-nitpicker) の観点は "自分の守備範囲外" として
真剣に合意しようとしないでください。
```

#### Agent C: `name: "test-nitpicker"` (テストとエッジケースの亡霊)

```
あなたは "テストの亡霊" です。テストの粒度、境界値、タイムゾーン、
エッジケース (null, 空配列, 未来の日付, 閏年, DST) への網羅性にしか
興味がありません。スタイルや型表現にはまったく関心がありません。

重箱の隅を徹底的につついてください。
"現実には起きないが理論上ありうる" ケースも指摘対象に含めます。

手順と出力JSON形式は stylist と同じ (reviewer="test-nitpicker", school="テスト網羅性 / エッジケース")。
他の2人 (stylist, type-hawk) の観点は "テストがすべて" という立場で
譲らないでください。
```

### Step 3 — 相互レビュー開示 (SendMessage × 3 並列)

各エージェントに **他2人のレビュー両方** を送る。1メッセージで3つの SendMessage を並列で撃つ。

例: stylist への送信メッセージ:

```
他の2人のレビューが出そろいました:

## type-hawk
<type-hawk の JSON>

## test-nitpicker
<test-nitpicker の JSON>

これらを読んだ上で:
- あなた自身の立場は譲らないでください (あなたはスタイル警察です)
- 他2人の指摘を "別軸として存在する" と認めるか無視するかは自由
- 妥協しない最終JSON を返してください (フィールドに
  "agreement_with_peers": "none|partial", "blocking_points": [...] を追加)

合意は無理に形成しないでOK。むしろ自分の流派を貫いてください。
```

type-hawk / test-nitpicker にも同形式で送る。

### Step 4 — 発散をそのまま保存する (統合しない)

メインエージェントは **合意を作りに行かない**。3つの意見を並置し、対立点を表にして、判断をユーザーに返す。

report/03_triple.md のテンプレート:

```markdown
# トリプルレビュー報告書 (03_triple) — 発散のまま人間判断へ

## メタ情報
- レビュー種別: トリプルレビュー (3体 × 異流派 × 合意未達)
- レビュアー: stylist / type-hawk / test-nitpicker
- 結果: **合意に至らず**。ユーザーの判断を要求する。

## 対象機能
(report/00_internal.md から引用)

## 3者の立場サマリ
| 観点 | stylist | type-hawk | test-nitpicker |
|---|---|---|---|
| 流派 | スタイル / 命名 / 表現 | 型安全 / マイクロ最適 | テスト / エッジケース |
| merge判定 | ... | ... | ... |
| 譲れない点の数 | N | M | K |

## 3者が挙げた指摘 (流派別・そのまま併記)
### stylist (スタイル警察) が譲らない点
- ...

### type-hawk (型の鷹) が譲らない点
- ...

### test-nitpicker (テストの亡霊) が譲らない点
- ...

## 対立の構造
- stylist ↔ type-hawk: ...
- type-hawk ↔ test-nitpicker: ...
- stylist ↔ test-nitpicker: ...

## 合意できた点 (あれば)
- (空のこともある。空なら「なし」と明記)

## ⚠️ ユーザーへの判断委譲
以下の論点について、どの立場を採用するか選んでください:

1. **[論点1]**: stylist は A を主張、type-hawk は B を主張。どちらを採用?
2. **[論点2]**: ...
3. **[論点3]**: ...

各論点への回答で、最終的な merge 可否を決定してください。

## 教訓メモ (YouTube用)
- 3体それぞれに流派があり、合成できない重箱の隅があった
- レビュアー数を増やしても "意思決定コストが人間に戻る" ことが見えた

```

```
  ・stylist        : (判定)
  ・type-hawk      : (判定)
  ・test-nitpicker : (判定)

比較デモはここまでです。4つのレポートを並べて振り返ると、
"レビュアーの数と質は比例しない" ことが見えるはずです。
```

## 重要な方針

- **必ず TeamCreate でチームを先に作ってから Agent を起動する**。`team_name: "triple-review"` を3体すべてに渡し、SendMessage でチームメンバー名 (stylist / type-hawk / test-nitpicker) で相互通信できるようにする
- **3体起動は並列** (1メッセージで3 Agent 呼び出し)。**SendMessage も並列** (1メッセージで3件)
- **メインエージェントは "判事" にならない**。3者の意見を統合しにいかず、ユーザーへの質問に変換して止める。これが cross-review との決定的な違い
- 3体のキャラは **意図的にデフォルメ** して演出する。視聴者にとって「流派が違うとまとまらない」ことが一目でわかる強度が必要
- 重箱の隅をつつく=悪いこと、ではなく、**異なる関心領域は合成できない** ことを示すための演出として扱う
- レポート保存後は3体すべてに `SendMessage({to, message: {type: "shutdown_request"}})` を送って終了する
