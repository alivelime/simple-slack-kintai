# self-review — セルフレビュー (最速・同一コンテキスト)

このスキルは、YouTube のAI駆動開発チャンネルで「クロスレビューの効率的なやり方」を比較デモする際の **1本目** として使う。視聴者にとっての分かりやすさが最優先。

## 目的

メインエージェントが機能を実装し、**実装した直後、同じコンテキストのまま自分でレビューする**。後続の single-review / cross-review / triple-review と対比できるよう、**速度と同一コンテキストゆえのバイアス** を際立たせる。

## 引数

ユーザーは `/self-review <機能の説明>` の形で呼び出す。例:

- `/self-review 月次の打刻データをCSVエクスポートする管理者向けAPI`
- `/self-review ダッシュボードに前月との勤務時間差分を表示`

引数が空のときは、ユーザーに何を実装すべきか確認する。

---

## 実行手順

### Step 1 — イントロ表示 (YouTube視聴者向け)

**実装やレビューの前に、必ず以下のブロックをそのまま出力する**。視聴者がこのスキルのポジショニングを一目で理解できるようにするための演出。

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎬 Pattern 1 / 4 : セルフレビュー
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【前提条件】
・これから機能を実装する(=まだレビュー対象は存在しない)
・レビュアーは実装者本人 (=同一コンテキスト)

【実行内容】
1. 指定された機能を実装する
2. 同じコンテキストのまま自分でレビューする
3. 結果を report/00_internal.md に保存する

【期待される効果】
✓ 実装直後なので文脈を完全に把握している → 速い
✓ 追加のトークン消費がほぼない → 安い

【このパターンの弱点】
✗ 自分の実装を自分で採点 → 確証バイアスが残る
✗ 「書いた時に見えなかった欠陥」はレビューでも見えない
✗ 実装方針そのものへの疑問は出にくい

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Step 2 — 機能を実装する

プロジェクトの CLAUDE.md と既存コードを踏まえて、引数で指定された機能を実装する。

- 既存のディレクトリ構造・コーディング規約に従う
- 実装は **YouTube尺に収まる現実的な粒度** で済ませる。過剰に広げない
- 実装の途中経過を視聴者にわかるよう、1〜2文のステータス更新を挟む
- ビルドや lint (`npm run lint`) を軽く走らせて、致命的エラーがないことだけは確認する

### Step 3 — 同一コンテキストでセルフレビューする

**サブエージェントは立てない**。このパターンの本質は「実装者がそのまま自己点検する」こと。

観点:

1. 実装した機能要件を満たしているか
2. 型の整合性、明らかなバグ、例外処理の漏れ
3. プロジェクトのコーディング規約との整合
4. 改善余地（自分で気づける範囲）

**正直さの演出**: このパターンは確証バイアスが残ることそのものが教訓なので、レビューは不自然に辛口にしない。実装者視点で気づける範囲を「正直に」書く。「自分で書いたコードだから大筋OKに見える」感覚を温存したまま書くのが、あとの比較で効いてくる。

### Step 4 — report/00_internal.md を保存する

以下のテンプレートに従う。

```markdown
# セルフレビュー報告書 (00_internal)

## メタ情報
- レビュー種別: セルフレビュー (同一コンテキスト)
- 実装者 / レビュアー: メインエージェント (同一)
- 所要時間の感覚: (速い / 普通 / 遅い のいずれか)
- バイアス注意: **確証バイアスあり**。実装直後の自己採点である点に注意。

## 対象機能
(引数で指定された機能の1〜3文サマリ)

## 実装した変更
- (変更ファイル1): 何をしたか
- (変更ファイル2): 何をしたか
...

## セルフレビュー所見
### 良い点 / 満たしている点
- ...

### 気になる点 / 改善余地
- ...

### 自分では判断できない点
- (客観的な第三者視点がほしい箇所を正直に列挙)

## このレビューで "見えていない可能性が高いもの" (自覚)
- (自分の思い込みがありそうな領域を挙げておくと、後続のsingle/cross/triple reviewとの比較が鮮明になる)

## 結論
(この実装を merge してよいかの自己判断。一言)
```

### Step 5 — 終了時のサマリ

最後に以下のサマリを表示する (視聴者向けの "まとめ")。

```
✅ Pattern 1 完了: セルフレビュー

レポート: report/00_internal.md
所要: (体感の速さ)
次のステップ: /single-review を実行して、独立したエージェントに同じコードをレビューさせる
```

---

## 重要な方針

- **凝りすぎない**。このスキルは「最速・最軽量」の見本なので、実装もレビューもシンプルに保つ
- **サブエージェントを絶対に呼ばない**。single-review 以降との対比が崩れる
- **report/00_internal.md は後続スキルが読む入力にもなる**。対象機能の説明と変更ファイル一覧は必ず埋める
