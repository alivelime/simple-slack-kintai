# 独立シングルレビュー報告書 (01_single)

## メタ情報
- レビュー種別: 独立シングルレビュー (別コンテキスト × 1体)
- レビュアー: サブエージェント (実装コンテキスト非共有)
- 入力: 変更コード (`app/api/slack/route.ts` の diff) + `report/00_internal.md`

## 対象機能
> Slack の `/punch out` を日跨ぎでも使えるようにする。23:50 に出勤して翌 00:30 に退勤しても、「まだ出勤してない」と弾かれず、出勤側の業務日 (= 前日) に退勤時刻が記録されるようにする。
> (report/00_internal.md より引用)

## 独立レビューの所見

### 🔴 重大な指摘

#### R1. `in` 側に **退行 (regression)**: 同一日の打刻し直しで `punch_in` が上書きされる
新しい `in` 分岐は「未退勤 (`punch_out IS NULL`) の最新レコード」しか見ていない。そのため以下のシーケンスで既存の確定済み出勤時刻を破壊できる。

1. 4/27 09:00 `/punch in` → `(date=2026-04-27, punch_in=09:00, punch_out=NULL)`
2. 4/27 18:00 `/punch out` → `(date=2026-04-27, punch_in=09:00, punch_out=18:00)` (確定)
3. 4/27 19:00 `/punch in` (打ち間違い) → 未退勤チェックは "punch_out IS NULL" の行しか見ないので **見落とす** → `upsert(onConflict: user_id,date)` が走り、**`punch_in=19:00` に上書き**。`punch_out=18:00` が残ったまま、退勤 < 出勤の不整合レコードが生まれる。

旧実装は「`today` の行に `punch_in` があれば一律拒否」だったので発生しなかった。今回の変更で **同日 2 回目 `/punch in` の防御が外れている**。`in` 側の事前チェックに「`today` で既に `punch_in` 済みのレコードを (退勤済みかどうかを問わず) 探す」を追加するか、`upsert` を `insert` に変えて UNIQUE 違反でエラーを返すべき。

セルフレビューはこの退行を完全に見落としている。

#### R2. `in` / `out` の組み合わせで **デッドロック状態** が成立しうる
- `MAX_OPEN_PUNCH_HOURS = 36` 超過の open punch があると `out` 側は「管理者に相談しな」で拒否。
- 一方 `in` 側は「未退勤 punch があれば一律拒否、out しろ」と返す。
- つまり 36h 超過の open punch が 1 件でも残っていると、ユーザーは `in` も `out` も叩けず、**管理者の手動修復まで打刻不能**。

セルフレビューでは「フロー的に微妙」と表現しているが、実際にはユーザー操作で抜け出せない閉じ込め (lock-out)。深夜シフトで偶発的に 36h を超えると即座に発動する。最低でも `in` 側は「36h 超過の古い open punch なら自動 cancel して新規 in を許す」「特殊文言で管理者対応を案内する」など、出口を用意すべき。

### 🟡 改善推奨

#### Y1. `out` の DB エラー時にロギングが無い
164–173 行で `update` が失敗してもユーザーに「失敗したぜ」と返すだけで `console.error` も Slack 通知もない。Slack コマンドはユーザー側ログが残らないので、何が起きたかが完全に不明になる。`error` を捕まえたら `console.error("[slack/out]", error)` を 1 行入れるだけで運用品質が大きく上がる。`in` の error 経路も同様。

#### Y2. 関数名 `jstNow()` と実装の不一致
`jstNow()` は `new Date().toISOString()` で **UTC ISO8601** を返している。命名と実装がずれており、レビュアー混乱の元。今回の変更で `new Date(now)` 比較に使っているので動作上は問題ないが、`utcNow()` 等にリネームするか、コメントで「TIMESTAMPTZ に流すための UTC 文字列」と明示するのが安全。今回の PR スコープ外でもよいが、独立観点として記録しておく。

#### Y3. 同時実行で 2 重 `out` が通る可能性
`select(open punch) → update by id` の間にもう 1 本の `/punch out` が同じ open punch を読むと、両者とも同じ `id` を更新する競合。実害は「punch_out が 2 番目の now で上書き」だけで小さいが、本気でやるなら `update ... where id=? and punch_out is null` の条件付き更新にして実際に変わった行数を確認するのが安全。Postgres であれば `.update().is("punch_out", null)` で十分。

#### Y4. `in` で「3 日前から未退勤」のときの文言が不親切
古い open punch の日付を生のまま `${openPunch.date}` で出すので、ユーザーが状況を理解しにくい。せめて経過日数を併記するか、「管理者に依頼してください」を必ず出すなど、(R2 と合わせて) 出口を文言レベルでも示すべき。

#### Y5. `closedToday` の追加クエリの位置ずれ
「open punch が無い」かつ「`today` で `punch_out` 済み」を二重退勤と判定しているが、`UNIQUE(user_id, date)` 制約があるため `today` のレコードはたかだか 1 件。それでも `eq("punch_out", null) is`-not みたいな分岐は `eq("date", today).single()` 1 回で十分で、`closedToday` 用に追加クエリを足したのは過剰。`select("punch_in, punch_out").eq("date", today)` の単一クエリで両ケースを判別できる。

#### Y6. `MAX_OPEN_PUNCH_HOURS` の境界値テスト不在 / 単位仕様の曖昧さ
`diffHours > 36` の境界に対して、ちょうど 36.0 のときどうするかが曖昧。深夜シフト規定によっては 36h ぴったりの勤務もありうる。`>=` か `>` か、業務要件で確定させるべき。コメントは「日跨ぎ退勤を許容する最大時間」だが、実際は「開始からの経過時間上限」で意味が違う。

#### Y7. インデックスのカバレッジ
`is("punch_out", null) AND not("punch_in", "is", null)` を含む新クエリは `idx_punch_records_user_date` (user_id, date) ではカバーされず、`user_id` の範囲スキャン後にフィルタになる。レコード数が増えてくると遅くなる。Slack コマンドは 3 秒タイムアウトなので、将来的には `WHERE punch_out IS NULL` 部分インデックスを足したい。今すぐではないが migration の TODO に。

### 🟢 良い点
- `single()` から `maybeSingle()` への置き換えは正しい。元コードは PGRST116 を握りつぶす形で動いていたので、今回の変更でクエリ意図が明確になった。
- `update` の絞り込みを `.eq("user_id").eq("date")` から **`.eq("id", openPunch.id)`** に切り替えた点は素直で、日跨ぎでの「業務日 = 出勤日」を維持する判断と整合している。
- 業務日ラベル `${openPunch.date} の勤務` を出すことで、ユーザーに「どの日の勤務に対する退勤か」が明示される。
- 西部劇口調・絵文字スタイルが既存と揃っている (CLAUDE.md の規約遵守)。
- `openPunch?.punch_in` のオプショナルチェーンを徹底しており型安全。

## セルフレビューとの差分

### セルフレビューが拾えていなかった点
- **R1: `in` 側で退勤済み同日レコードを見落として `punch_in` を上書きする退行**。ここがいちばん厳しい欠陥。セルフレビューでは「`in` 側にも未退勤チェックを入れたので親切」とポジティブに評価しているが、実際には旧実装の防御を弱めている。
- **R2: 36h 超過時に `in` も `out` も通らないデッドロック状態**。セルフレビューでは「フロー的に微妙」と緩く触れているだけで、ロックアウトの深刻さに踏み込んでいない。
- **Y1: DB エラー時のロギング欠落** — 観点として上がっていない。
- **Y2: `jstNow()` の名前と実装の不一致** — 元からのものとはいえ、独立レビューでは明白に違和感が出る。
- **Y3: `out` 同時実行の条件付き更新化** — 「同時実行は要件次第」と片付けているが、`update().is("punch_out", null)` でほぼ無料で防げる。
- **Y5: 追加クエリの単純化余地** — 「煩雑」と感じてはいるが具体策には踏み込んでいない。

### セルフレビューが過剰に楽観視していた点
- 「**`in` 側にも未退勤チェックを入れたので親切**」: 実際は退勤済み行を見ない一方で `upsert` を残しているため、防御が薄くなる方向に変わってしまっている (R1)。
- 「**36 時間ガードで暴発防止**」: 暴発は防げているが、その代償として「自分では復帰できない状態」を作っている (R2)。トレードオフの片面しか評価していない。
- 「**lint は元々ある 3 件のみ**」: lint が通ることと、ロジックの正しさは独立。

### セルフレビューと一致した点
- 業務日 = 出勤日というセマンティクスを維持している点 (ダッシュボード集計を壊さない) は正しく、独立レビューでも崩れていないことを確認した。
- `36` というマジックナンバーが業務要件未確認である点。
- 同時実行の整合性に懸念がある点 (本レビューでは Y3 として具体化)。
- テスト不在。

## 結論
**現状のまま merge は不可**。最低限 **R1 (同一日の `in` 上書き退行)** は本 PR 内で塞ぐべき。R2 (デッドロック) は文言だけでも出口を提示するか、`in` 側で 36h 超過 open punch を検出したら別文言で管理者連絡を促すのが望ましい。

優先度:
1. R1 を必ず修正 (同日の `punch_in` 既存チェックを `punch_out` の有無に関わらず行う / もしくは `upsert` を `insert` に切り替えて UNIQUE 違反をユーザーに伝える)
2. R2 を最低でも文言で緩和 (本格対応は要件確認後)
3. Y1 (DB エラーログ) は当 PR で入れてしまうのを推奨
4. Y3, Y5, Y6, Y7 は別 issue で OK
